### 1. MinIO 8.5.7 版本 API 签名全面变更，旧代码全部不可编译

**时间**：2026-05-05

**背景**：storage 模块需要实现 MinIO 分片上传。按照 Java 生态的普遍认知，S3 兼容的对象存储都有 `createMultipartUpload` / `uploadPart` / `completeMultipartUpload` 这样的方法，我直接按这个习惯写了，结果编译阶段直接报方法不存在。

**现象**：`StorageServiceImpl` 中调用 `minioClient.createMultipartUpload()`、`minioClient.uploadPart()` 等 5 个方法全部编译失败，IDE 提示 "The method is undefined"。同时 `interface` 作为 Java 包名被 IDE 解析为关键字，触发 "Syntax error on token 'interface'" 连锁报错。

**排查过程**：
1. 确认依赖版本：`pom.xml` 中声明的是 `minio:8.5.7`，jar 位于 `~/.m2/repository/io/minio/minio/8.5.7/minio-8.5.7.jar`
2. 用 `javap -public` 反编译 `MinioClient.class`，列出全部 public 方法
3. 发现 MinIO 在 8.5.x 版本已全面废弃旧式 S3 风格方法，只保留了 Builder 模式的 API：`putObject(PutObjectArgs)`、`composeObject(ComposeObjectArgs)`、`removeObject(RemoveObjectArgs)` 等，没有任何 `createMultipartUpload` 或 `uploadPart`
4. `composeObject` 支持传入多个源对象，在服务端将它们拼接为一个新对象——这正好可以替代 S3 multipart upload

**根本原因**：凭经验假设 API 而非真实验证。S3 风格的 multipart 上传方法在 MinIO Java SDK 的早期版本（7.x/8.0.x）确实存在，但 8.5.x 已经统一为 Builder 模式。同时 `interface` 是 Java 保留字，用作包名虽语法不报错但 IDE 静态分析时行为不确定。

**解决方案**：
1. 放弃 S3 multipart API，改为 composeObject 方案：每个分片作为独立对象上传到 `{md5}/chunks/00001`，全部上传完成后调用 `composeObject` 在 MinIO 服务端合并，再删除临时分片
2. `sed` 批量替换所有模块中 `package com.localrag.xxx.interface` → `package com.localrag.xxx.contract`

**收获**：引入新依赖前先 `javap` 看一眼 API 长什么样，不要凭经验脑补。Java 包名避开 `interface`、`class`、`new` 等保留字，用 `contract` 不但安全而且语义更准确。

### 2. Spring Boot 多模块工程首次启动的配置链问题

**时间**：2026-05-05

**背景**：项目是 Maven 多模块工程（10 个子模块），api 模块是唯一启动入口，依序依赖所有下层模块。首次尝试启动 api 模块。

**现象**：三个问题依次出现：
1. `mvn spring-boot:run -pl api` 报 `Could not find artifact com.localrag:common:jar:1.0.0-SNAPSHOT` —— 所有内部模块的 jar 都找不到
2. 修复后启动报 `Failed to bind properties under 'spring.web.resources'` —— 应用能启动但 Spring 容器初始化失败
3. 端口 8080 被其他项目占用

**排查过程**：
1. `mvn spring-boot:run -pl api` 的工作机制是：Maven 只解析 api 模块的依赖，然后去本地仓库 `~/.m2` 里找 jar。但其他 9 个子模块从未被 `install` 过，本地仓库里根本没有。
2. 看完整启动日志中的 `ConditionEvaluationReport`，Spring Boot 提示 `spring.web.resources` 属性绑定失败。逐行检查 `application.yml`，发现 `spring.web.resources:` 下面没有任何子属性——之前删 `static-locations` 时把值删了但父 key 还在。

**根本原因**：
1. Maven 多模块项目中，`-pl` 只是 "以这个模块为目标执行 goal"，但它的依赖解析仍然走本地仓库。`compile` 阶段只生成 `target/classes`，不会写 `~/.m2`，所以 `run` 找不到依赖 jar。
2. YAML 中空的 key 在 Spring Boot 的属性绑定阶段会被当作一个合法的配置节点（类型是 `String`），尝试转换到 `WebProperties$Resources` 类型时找不到对应的 Converter。

**解决方案**：
1. 先 `mvn clean install -DskipTests` 把所有模块的 jar 安装到本地仓库，再 `mvn spring-boot:run -pl api` 启动。`install` 是必须的中间步骤。
2. 删除空的 `spring.web.resources` 节点（Spring Boot 默认从 `classpath:/static/` 提供静态文件，不需要显式声明）
3. 端口改为 8088

**收获**：Maven 多模块项目 `-pl` 只决定 "编译谁"，不决定 "在哪找依赖"。依赖查找永远走本地仓库 + 远程仓库，所以底下模块必须先 `install`。YAML 配置坚持"不留空 key"原则。

### 3. ES 连接协议错误：代码用 HTTPS 但容器关了 SSL

**时间**：2026-05-05

**背景**：vector-store 模块写完后首次启动应用，`IndexManager` 在 `CommandLineRunner` 中自动创建 ES 索引。

**现象**：应用一启动就退出，报 `VectorStoreException: ES 索引创建失败`。

**排查过程**：
1. `docker ps` 确认 ES 容器是 Up 且 healthy → 排除容器没起的可能
2. 翻 `docker-compose.yml` 的 ES 配置，发现 `xpack.security.http.ssl.enabled=false` —— SSL 被显式关闭，HTTP 层是明文传输
3. 回到 `EsConfig.java`，看到 `new HttpHost(host, port, "https")`，客户端在跟 HTTP 端口做 SSL 握手，直接被拒绝

**根本原因**：Docker ES 配了 `xpack.security.http.ssl.enabled=false`，认证仅靠 basic auth（用户名密码），不依赖 SSL。代码写了 HTTPS，协议不匹配。

**解决方案**：
1. `EsConfig` 中改为 `new HttpHost(host, port, "http")`
2. `IndexManager.run()` 外层加 try-catch，ES 不可用时打 WARN 不阻塞启动

**收获**：代码里的连接参数（协议/端口/认证方式）必须与 `docker-compose.yml` 逐项对齐，不能靠"默认值"猜测。

### 4. Redis 脏数据：上次上传中断，残留不完整 Hash 导致 parseInt 崩溃

**时间**：2026-05-05

**背景**：修复 ES 后重新启动，第一次测试上传文件。

**现象**：浏览器选文件 → 点上传 → InitUpload 返回 500。后台日志 `NumberFormatException: For input string: ""`，位置在 `RedisUploadStateManager.getByMd5` 的 `Long.parseLong(getStr(fields, "fileSize"))`。

**排查过程**：
1. `curl -X POST /api/storage/upload/init` 复现问题，堆栈一致指向 `Long.parseLong("")`
2. 进 `getByMd5` 看逻辑：从 Redis 读到 key 存在，但 `fileSize` 字段的值是空字符串
3. 追溯来源：上一次测试时 init 调用成功执行了 `tryCreate()` 写入 Redis，但后续步骤（application restart / ES 崩溃）中断了流程，Redis 里的 Hash 数据是残缺的——key 存在，但字段缺了几个

**根本原因**：`getByMd5` 只检查了 Redis key 是否存在（`fields.isEmpty()`），没有逐字段校验值是否合法。类似于数据库查回来一行就假设所有列都有值，但实际某列可能是 NULL。

**解决方案**：增加字段完整性校验——`fileSize` / `totalParts` / `status` 任一为空字符串 → 视为脏数据 → 自动删除 Redis key → 返回 null 走正常的新上传流程。外层再加 try-catch 兜底。

**收获**：从外部存储读回来的数据，永远做防御性校验。程序可能在写入中途崩溃、Redis 可能淘汰了部分字段、也可能被人手动改过——不能假设"刚写进去的所以一定完整"。

### 5. Map.of() 空值崩溃：uploadId 字段 refactor 后变为 null，触发不可变集合的 null 检查

**时间**：2026-05-05

**背景**：修复 Redis 脏数据后再次测试上传。之前在 issue #1 中做过一次全局 refactor，把 MinIO S3 风格 API 改成了 composeObject 方案，不再需要 MinIO 的 `uploadId`。`StorageService` 接口删了 `createMultipartUpload` 方法，`StorageController` 也不再调用它——修改涉及 3 个文件。

**现象**：InitUpload 又一次 500，后台 `NullPointerException`，堆栈：

```
java.lang.NullPointerException: null
    at java.base/java.util.Objects.requireNonNull(Objects.java:209)
    at java.base/java.util.ImmutableCollections$MapN.<init>(ImmutableCollections.java:1186)
    ...
    at com.localrag.storage.impl.RedisUploadStateManager.tryCreate(RedisUploadStateManager.java:37)
    at com.localrag.controller.StorageController.initUpload(StorageController.java:89)
```

关键线索：`ImmutableCollections$MapN.<init>` —— Java 不可变 Map 的构造函数，由 `Map.of()` 调用触发。

**排查过程**：
1. 堆栈精确定位到 `RedisUploadStateManager.tryCreate()` 中的 `Map.of(...)`
2. `Map.of()` 的 Javadoc 明确写：`@throws NullPointerException if any key or value is null`。内部对每个 key 和 value 调用 `Objects.requireNonNull()`
3. 逐项排查 `Map.of()` 传入的 8 个参数值：md5 ✅、fileName ✅、fileSize ✅、uploadId ❌ → null
4. 回溯 null 的来源：`task.getUploadId()`。`UploadTask` 类中 `uploadId` 字段没有 `@Builder.Default`，默认就是 null。之前的 refactor 在 `StorageController.initUpload()` 里移除了 `.uploadId()` 的 Builder 赋值行，但 `tryCreate()` 里 `Map.of("uploadId", ...)` 这一行没被同步删掉

**根本原因**：一次 refactor 改了接口、实现、Controller 三层，但漏了 `tryCreate()` 中的 `Map.of("uploadId", ...)`。`Map.of()` 设计用于编译期确定的常量集合（如 `Map.of("key1", val1, "key2", val2)`），它假定所有值非 null。当 value 来自一个业务对象的可能为 null 的字段时，这个假定不成立。

**解决方案**：
1. `Map.of()` 改回 `new HashMap<>()` + `put()`——传统写法接受 null，而且语义上我们就是在"动态构建一个 Map"，不是"声明编译期常量"
2. 既然 composeObject 方案不需要 uploadId，直接从 Map 中去掉这个字段

```java
// 改之前：Map.of() + uploadId 为 null → NPE
Map<String, String> fields = Map.of(
    "md5", task.getMd5(),
    "fileSize", String.valueOf(task.getFileSize()),
    "uploadId", task.getUploadId(),  // ← null
    "bucket", task.getBucket(),
    ...
);

// 改之后：HashMap + 去掉 uploadId
Map<String, String> fields = new HashMap<>();
fields.put("md5", task.getMd5());
fields.put("fileSize", String.valueOf(task.getFileSize()));
fields.put("bucket", task.getBucket());
...
// uploadId 不再写入，composeObject 方案不需要
```

**收获**：
- **refactor 后必须全局 `grep` 被删字段的引用**：删掉一个字段的赋值之后，搜索整个代码库看看还有哪里在读取或使用它。不能只改了 3 个文件就认为改完了。
- **`Map.of()` / `List.of()` 仅适用于编译期常量场景**：如配置文件映射、错误码表、固定枚举对应关系。如果 value 来自方法参数、对象字段或网络数据——它们都有可能为 null——用传统 HashMap / ArrayList 更安全。
- **不可变集合的 null 检查发生在构造阶段而非使用阶段**：`Map.of("key", null)` 在调用时就崩，不是等到后续 `get("key")` 时才崩。排查时看堆栈最上面几行（`ImmutableCollections$MapN`）就能直接定位。

### 6. DeepSeek SSE 流式响应中文乱码 + API Key 分层配置遗漏 + 聊天记录表迁移

**时间**：2026-05-06

**现象**：
1. 对话页面提问后返回的中文全部乱码（`浣犲ソ` 等）
2. 提问后无任何响应（API 返回 401 或静默失败）
3. 页面切换回来聊天记录消失或重复出现

**造成原因**：
1. `InputStreamReader(response.body().byteStream())` 未指定字符集，Windows 默认用 GBK 解码 UTF-8 数据，导致中文乱码
2. `application-local.yml` 只配了 `embedding` 的 API Key，漏了 `llm` 的 Key。`LlmConfig` 读 `${DEEPSEEK_API_KEY:}` 取到空字符串，DeepSeek API 返回 401
3. `loadHistory()` 每次调用只追加 DOM 不清空旧内容，页面切换后历史重复累积

**解决方法**：
1. `new InputStreamReader(stream, StandardCharsets.UTF_8)` 显式指定编码
2. `application-local.yml` 补全 `localrag.llm.deepseek.api-key` 配置项
3. `loadHistory()` 开头加 `document.getElementById('chatBox').innerHTML = ''`
4. `ChatHistoryMessage` 表新增 `message_id`(UUID) 和 `user_id` 字段，旧表需 DROP 重建

**收获**：
- Java IO 操作永远显式指定字符集，不依赖平台默认编码
- 多模块项目每个模块的配置项要逐个确认，不能假设"之前配过所以这次也是好的"
- SSE 流式接口排查时先 `curl -v` 看 HTTP 状态码和响应头，再检查 InputStream 编解码

### 7. `@Transactional` 注解用错包导致删除操作静默失效

**时间**：2026-05-06

**现象**：用户点击删除会话，前端调 `DELETE /api/llm/sessions/{id}` 返回 200，刷新页面数据还原样存在，MySQL 数据一条没少。不报错、不抛异常、日志正常——就是删不掉。

**造成原因**：`ChatHistoryRepository.deleteBySessionId()` 上标注了 `@Transactional`，但导的包是 `jakarta.transaction.Transactional`，Spring 的事务管理器不认 Jakarta 的注解，等于没开事务。JPA 的 derived delete 方法在没事务包裹时，Hibernate 不会自动 flush 和 commit，delete 操作被无声丢弃。

Spring Boot 3.x 迁移到 Jakarta EE 后，`jakarta.transaction.Transactional` 和 `org.springframework.transaction.annotation.Transactional` 两个注解名字几乎一样，IDE 自动补全容易选错。

**解决方法**：换成 `org.springframework.transaction.annotation.Transactional`，同时去掉多余的 `@Modifying`（Spring Data JPA 的 derived delete 不需要显式声明 `@Modifying`，它只用于自定义 `@Query` 注解的 UPDATE/DELETE 语句）。

**收获**：Spring Boot 3.x 项目中注意区分 Jakarta 和 Spring 的同名注解——`@Transactional`、`@Inject`、`@PostConstruct` 都有两个版本。排查"接口返回成功但数据没变"的问题时，第一时间检查事务配置。

### 8. Fetch API 消费 SSE 流不如 EventSource，导致流式对话卡死不刷新

**时间**：2026-05-06

**现象**：LLM 对话用 Fetch API 的 `response.body.getReader()` 消费后端 SSE 流，页面卡住不动，只有切换页面再回来才能看到完整回答——完全没有"逐字输出"的流式效果。试了 `setTimeout(read, 0)` 让步浏览器重绘、`SseEmitter` 手动 `flush()` 都无效。

**造成原因**：Fetch API 的 `ReadableStream` reader 是通用流式 API，递归调用 `reader.read()` 时浏览器事件循环被密集占用，没有间隙触发 DOM 重绘。而 `EventSource` 是 W3C 专为 SSE 设计的标准 API，浏览器内部自动将 `data:` 行的解析和 `onmessage` 回调编排到事件循环中，每收到一行触发一次回调并自然留出重绘窗口。

**解决方法**：后端改为 `GET /api/llm/chat?query=xxx&sessionId=xxx`（EventSource 只支持 GET，不支持 POST），返回 `Content-Type: text/event-stream`。前端改为 `new EventSource(url)` → `es.onmessage` 逐字追加 DOM。

**收获**：SSE 场景优先用 `EventSource`，它是浏览器为 SSE 做的原生实现，自动处理解析、重连、事件循环协调。`Fetch` + `ReadableStream` 适用于非 SSE 的流式场景（如文件下载进度），不要套在 SSE 上。

### 9. 文本分块参数理解偏差——bufferSize 当成块大小，实际产出 5000 字/块

**时间**：2026-05-06

**现象**：PDF 上传后只切出 23 块，每块约 5000 字，导致检索时语义粒度过粗——"IO多路复用" 的内容被埋在一个大块中，KNN 向量检索没排到 Top 30，用户搜索返回"不存在相关信息"。

**造成原因**：分块器 `FixedSizeTextChunker` 的参数 `bufferSize` 被误当作"窗口大小"。调用方传 `bufferSize=5000`，chunker 按 5000 字符截取一个窗口，窗口内再按段落/句子边界输出块。结果每块也是 ~5000 字，而非预期的 ~512 字。代码逻辑和参数命名都没错，但参数值 5000 偏离了业务需求。

**解决方法**：`bufferSize` 改为 512，`minChunk` 改为 128。同时简化策略：段落 < 512 字直接成块，段落 ≥ 512 字按句子切分，句子太长走 HanLP 分词保底，所有块 ≤ 512 字。相邻块 15% 重叠防止语义断裂。

**收获**：分块参数要和"向量化后检索的语义粒度"挂钩——太大则检索不精确，太小则语义碎片化。新增一个参数时要明确它是"目标值"还是"触发阈值"，避免命名歧义。参数调完后必须删旧数据重跑，否则新旧块混在一起。

### 10. 切细块后 Kafka 消息体积超限——194 块打包一条消息 5MB 触发 RecordTooLargeException

**时间**：2026-05-06

**现象**：切块参数从 5000 改为 512 后，同一个 PDF 从 23 块变成了 194 块。DocumentConsumer 把所有块打包成一条 `document.chunked` Kafka 消息发送，序列化后约 5MB，远超 Kafka 默认 `max.request.size=1MB`，Kafka Producer 直接抛 `RecordTooLargeException`，消息发送失败，文档无法进入下游处理。

**造成原因**：之前的代码假设"一个文件 → 几十个块 → 一条 Kafka 消息"，没有对每批消息做体积限制。当块数从 23 涨到 194，单条消息体积从 ~500KB 暴涨到 ~5MB。

**解决方法**：DocumentConsumer 和 EmbeddingConsumer 各增加分批发逻辑——每 20 个 chunk 打包一条 Kafka 消息。194 块分成 10 条 `document.chunked` 消息，每条约 200KB，远低于 1MB 限制。EmbeddingConsumer 同理，输出 `embedding.requested` 时也每 20 个向量化 chunk 发一条。

**收获**：消息队列场景中，单条消息体积上限是硬约束。必须在生产者侧做分页/分批，不能依赖下游"反正都消费"的假设。分块策略和消息体积之间的矛盾——块小了检索精确但消息数量暴增，需要在 chunk 数量和单消息 chunk 数之间取平衡。
